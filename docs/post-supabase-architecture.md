# Post-Supabase Architecture Guide

Last updated: 2026-03-05

## Purpose
This document is the internal source of truth for the current Midday architecture after moving away from Supabase-managed runtime dependencies.

Target runtime stack:
- Auth: Zitadel (OIDC + JWT)
- Relational DB: PostgreSQL (self-hosted / managed outside Supabase)
- File storage: AWS S3-compatible object storage
- Realtime: internal WebSocket contract + client polling fallback

This migration assumes a greenfield cutover (no production user/data migration constraints).

## High-Level System
Core applications and roles:
- `apps/dashboard` (Next.js): web UI, login/logout/callback, user-facing uploads, realtime subscriptions.
- `apps/api` (Hono + tRPC + REST): authn/authz boundary, business logic, signed URL generation, file serving, API key/OAuth token support.
- `apps/worker` (BullMQ): async processing (documents, invoices, banking, notifications).
- `packages/db` (Drizzle): schema, migrations, query layer.
- `packages/auth`: Zitadel OIDC/JWT/session helpers.
- `packages/storage`: S3 abstraction (upload/download/remove/signed URLs/public URLs).
- `packages/realtime`: shared event envelope contracts + publisher abstraction.

## Auth Architecture (Zitadel)

### Dashboard login/session flow
Key files:
- `apps/dashboard/src/app/api/auth/login/route.ts`
- `apps/dashboard/src/app/api/auth/callback/route.ts`
- `apps/dashboard/src/app/api/auth/logout/route.ts`
- `packages/auth/src/oidc.ts`
- `packages/auth/src/session.ts`

Flow:
1. `/api/auth/login` creates PKCE verifier/challenge and `state`, stores auth-flow cookies, redirects to Zitadel authorize endpoint.
2. `/api/auth/callback` validates `state`, exchanges `code` for tokens, verifies access token claims, and issues app session cookies.
3. Access token is stored in `midday-access-token` cookie (used for API bearer auth).
4. App session JWT (`midday-auth-session`) is signed by `APP_SESSION_SECRET`/`AUTH_SESSION_SECRET`.
5. Optional MFA enforcement in dashboard middleware (`ZITADEL_ENFORCE_MFA=true`) checks `amr` in token claims.

Cookie names are centralized in:
- `apps/dashboard/src/utils/constants.ts`

### API token validation and user resolution
Key files:
- `apps/api/src/utils/auth.ts`
- `apps/api/src/rest/middleware/auth.ts`
- `apps/api/src/trpc/init.ts`
- `packages/auth/src/jwt.ts`

Flow:
1. API receives bearer token.
2. `verifyAccessToken` validates against Zitadel JWKS (`ZITADEL_ISSUER`, optional `ZITADEL_AUDIENCE`).
3. API resolves internal user by `auth_subject` (`users.auth_subject`), with bootstrap upsert on first login.
4. API also still supports:
- internal OAuth access tokens (`mid_access_token_*`)
- API keys (`mid_*`)

Identity mapping:
- Migration: `packages/db/migrations/0033_add_users_auth_subject.sql`
- Queries: `packages/db/src/queries/users.ts`

## PostgreSQL Architecture
Key files:
- `packages/db/src/schema.ts`
- `packages/db/src/queries/*`
- `apps/api/.env-template` (DB envs)

Current model:
- API reads/writes via Drizzle query layer (`@midday/db/queries`).
- Job workers use `createJobDb()` with constrained pool settings for background load control.

Primary envs:
- `DATABASE_PRIMARY_URL`
- `DATABASE_FRA_URL`
- `DATABASE_SJC_URL`
- `DATABASE_IAD_URL`
- Worker/job path uses `DATABASE_PRIMARY_POOLER_URL`

## Storage Architecture (S3)

### Storage package contract
Key file:
- `packages/storage/src/index.ts`

Provided functions:
- `upload`
- `download`
- `remove`
- `exists`
- `createSignedUrl`
- `createPresignedUploadUrl`
- `createPublicUrl`

Primary envs:
- `AWS_REGION`
- `S3_BUCKET`
- `S3_ENDPOINT` (optional)
- `S3_FORCE_PATH_STYLE`
- `S3_PUBLIC_BASE_URL` (optional)
- `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` / `AWS_SESSION_TOKEN` (if not using ambient IAM)

### Upload + signed URL flow
Key files:
- `apps/dashboard/src/utils/upload.ts`
- `apps/dashboard/src/app/api/storage/upload-url/route.ts`
- `apps/api/src/trpc/routers/documents.ts`
- `apps/api/src/rest/routers/documents.ts`

Flow:
1. Dashboard requests presigned PUT URL from `/api/storage/upload-url`.
2. Route proxies request to tRPC `documents.createUploadUrl`.
3. API returns presigned S3 URL + public URL metadata.
4. Browser uploads directly to S3 via `XMLHttpRequest` PUT.
5. Download and access links are generated by API using signed URLs.

## Realtime Architecture
Key files:
- `packages/realtime/src/contracts.ts`
- `packages/realtime/src/publisher.ts`
- `apps/dashboard/src/hooks/use-realtime.ts`

Current behavior:
- Shared envelope format: `RealtimeEnvelope` / `RealtimeEvent`.
- Dashboard subscribes over WebSocket (`NEXT_PUBLIC_REALTIME_URL` or derived from `NEXT_PUBLIC_API_URL`).
- On disconnect/close, dashboard falls back to polling callback every 15s.
- Publish transport is pluggable via `setRealtimePublishTransport(...)`.

Current status note:
- Contracts and client transport are in place.
- Publisher transport wiring (server-side fanout implementation) is expected to be connected in the API/service runtime.

## Compatibility Layers (Transitional)
Even after runtime migration, compatibility shims still exist for safe incremental cleanup:
- `apps/api/src/services/storage-client.ts`

Behavior:
- Storage calls are backed by `@midday/storage` (S3).
- Legacy table-style `.from(...)` access is intentionally blocked with explicit errors.
- Legacy admin delete-user path is currently a safe stub; full Zitadel management API deletion can be wired later.

## Environment Matrix (Minimum by service)

### API (`apps/api/.env-template`)
Required minimum:
- `ZITADEL_ISSUER` (`ZITADEL_AUDIENCE` recommended)
- `DATABASE_PRIMARY_URL`
- `S3_BUCKET`, `AWS_REGION`

### Dashboard (`apps/dashboard/.env-example`)
Required minimum:
- `ZITADEL_ISSUER`, `ZITADEL_CLIENT_ID`
- `NEXT_PUBLIC_URL`, `NEXT_PUBLIC_API_URL`
- `NEXT_PUBLIC_REALTIME_URL` (recommended explicit)
- `APP_SESSION_SECRET`

### Worker (`apps/worker/.env-template`)
Required minimum:
- `DATABASE_PRIMARY_POOLER_URL`
- `ZITADEL_ISSUER` (if validating access tokens in worker paths; `ZITADEL_AUDIENCE` recommended)
- `S3_BUCKET`, `AWS_REGION`
- `REDIS_QUEUE_URL`

## Remaining Supabase-Named Artifacts

- docs/website historical posts and README mentions

## Operational Checks
Use these checks during rollout:

1. Verify no active runtime imports:
```bash
rg -n "@supabase/|@midday/supabase" apps/api apps/dashboard apps/worker packages --glob '!**/*.md' --glob '!**/*.mdx'
```

2. Verify Zitadel config is present:
```bash
rg -n "ZITADEL_" apps/api/.env-template apps/dashboard/.env-example apps/worker/.env-template
```

3. Verify S3 config is present:
```bash
rg -n "S3_|AWS_REGION" apps/api/.env-template apps/dashboard/.env-example apps/worker/.env-template
```

4. Verify auth subject migration exists:
```bash
ls packages/db/migrations/0033_add_users_auth_subject.sql
```
